# AI 发现候选 Key → 用户确认 → add_key 交互设计

> 状态：设计阶段

## 场景

AI 在帮用户写代码或排查问题时，可能发现用户的环境中存在可以沉淀的 key，但当前不会自动入库。

本设计描述如何在保持用户控制权的前提下，让 AI 参与密钥发现。

## 当前边界（不变）

- 不会因为 AI 对话里出现 token 就自动入库
- `scan` 默认只预览，只有 `--apply` 或交互确认才会导入
- `reuse_add_key` 是 MCP 工具，AI 需要显式调用，不会静默触发

## 推荐交互流程

```
1. AI 发现候选（被动）
   ┌─────────────────────────────────────┐
   │ AI 在上下文中发现用户提到或使用了   │
   │ 一个 API key / token / secret       │
   └──────────────┬──────────────────────┘
                  ▼
2. AI 向用户确认
   ┌─────────────────────────────────────┐
   │ "我注意到你在用 RESEND_API_KEY。    │
   │  要我把它存到 KeyFlow 吗？"         │
   │                                     │
   │  provider: resend                   │
   │  projects: marketing-site           │
   │  account: acme-mail                 │
   └──────────────┬──────────────────────┘
                  ▼
3. 用户确认
   ┌─────────────────────────────────────┐
   │ 用户："好的，存起来"                │
   └──────────────┬──────────────────────┘
                  ▼
4. AI 调用 add_key
   ┌─────────────────────────────────────┐
   │ MCP: add_key(                       │
   │   env_var: "RESEND_API_KEY",        │
   │   value: "re_xxx",                  │
   │   provider: "resend",               │
   │   account_name: "acme-mail",        │
   │   projects: ["marketing-site"]      │
   │ )                                   │
   └─────────────────────────────────────┘
```

## AI 侧实现建议

MCP 工具不需要新增。当前 `reuse_add_key` + `search_keys` 已够用。

AI 侧的行为准则（写在 MCP server 的 tool description 或系统 prompt 中）：

1. **发现** — 如果用户在对话中粘贴了一段 key，或者在 .env 中看到了 key
2. **确认** — 先问用户是否要存，不要直接调 add_key
3. **补全** — 如果能推断 provider / account / project，主动补上
4. **入库** — 用户确认后再调 add_key

## 不应该做的事

- 不要扫描用户终端输出并自动保存
- 不要把对话历史中出现的所有疑似 key 都存进去
- 不要在用户没确认时调用 add_key
- 不要保存明显是示例或文档中的 key（sk-xxx, YOUR_API_KEY）

## 与 `kf scan` 的关系

`kf scan` 是 CLI 侧的批量发现工具，面向文件系统。
本设计是 AI 侧的单条发现流程，面向对话上下文。

两者互补，不冲突。
